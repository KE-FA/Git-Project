What is Git?

Git is a free and open source distributed version control system designed to handle everything from small to very large projects with speed and efficiency.

Git as a Version Control System

A Version Control System is a tool that helps manage changes to files over time. It allows multiple users to collaborate on projects while keeping track of every modification made to the code or documents.

Other Version Control Systems

  • Subversion (SVN)
  • Mercurial
  • CVS

Configuring Git

After installing Git, you need to configure it with your name and email address. This information will be associated with your commits. You can set this up using the following commands in your terminal or command prompt: 

  • git config --global user.name "Your Name"

    • Replace "Your Name" with your actual name.

  • git config --global user.email "Your Email"

    • Replace ""Your Email" with your actual email address.

  • You can verify your configuration by running the following command: 

    • git config --list

    This will display your Git configuration settings, including your name and email address.

  • You can also configure the editor using the following:

    • git config --global core.editor "your_editor"

    Replace "your_editor" with the name of your preferred text editor (e.g., nano, vim, code, etc.).

Intializing a Git repository

First verify if the git repository is present by using the following command: 

  • git status

    • If you see a message like "fatal: not a git repository (or any of the parent directories): .git", it means you are not in a Git repository.

If the repository is not present you can initialize it using the following command: 

  • git init

    • This will create a new subdirectory named .git that contains all of your necessary repository files. You can verify that the repository has been initialized by checking for the presence of the .git directory.

Adding files to the staging area

The staging area is a file, generally contained in your Git directory, that stores information about what will go into your next commit.

Tracked Vs Untracked Files

  • Tracked files are files that Git already knows about—they've been added and committed at least once. Changes to these files will show up when you run git status.

  • Untracked files are new files in your working directory that Git isn't yet monitoring. These files won’t be included in commits unless you explicitly add them.

Adding files to the staging area

To add files to the staging area,you can use the following command:

  • git add filename

    • Replace "filename" with the name of the file you want to add.

  • You can also add all files in the current directory and its subdirectories using the following command: 

    • git add .

    This will stage all changes (new files, modified files, and deleted files) in the current directory.

Commiting changes

Making a Commit

After adding files to the staging area, you need to commit them to the repository.A commit is a snapshot of your changes at a specific point in time.

To commit changes, use the following command:

  • git commit -m "Your commit message"

    • Replace "Your commit message" with a brief description of your changes.

  • You can also include a descriptive message using the command:

    • git commit

    This will open your default text editor, allowing you to write a more detailed commit message.

Writing Good Commit Messages

A good commit message should be concise yet descriptive. It should explain what changes were made and why they were made. Here are some tips for writing good commit messages:

  • Use the imperative mood (e.g., "Fix bug" instead of "Fixed bug").
  • Keep it short and to the point.
  • Provide context for your changes.
  • Use bullet points for longer messages.

Viewing Past Commits

You can view the history of commits in your repository using the following command: 

  • git log

    • This will display a list of commits, along with their commit messages.

    You can also use the following command to view a more concise log:

  • git log --oneline

    • This will display a one-line summary of each commit, making it easier to read.

Going Back to a Previous Commit

If you need to revert to a previous commit, you can use the following command: 

  • git checkout commit_hash

    • Replace "commit_hash" with the hash of the commit you want to revert to. You can find the commit hash using the git log command.

    This will switch your working directory to the specified commit, allowing you to view the state of the repository at that point in time.

Branches

A branch in Git is essentially a pointer to a snapshot of your changes. It lets you work on new features, bug fixes, or experiments without affecting the main project code

Creating and Viewing Branches

To create a new branch, use the following command: 

  • git branch branch_name

    • Replace "branch_name" with the name of your new branch.

  • Using the git branch will list all branches as shown in the figure above.

The currently checked-out branch will have an asterisk (*) next to its name.

Switching Branches

To switch to a different branch, use the following command: 

  • git switch branch_name

    • Replace "branch_name" with the name of the branch you want to switch to.

Merging Branches

To merge changes from one branch into another, first switch to the branch you want to merge into, and then use the following command: 

  • git merge branch_name

    • Replace "branch_name" with the name of the branch you want to merge.

This will merge the changes from the specified branch into your current branch.

Deleting Branches

To delete a branch that you no longer need, use the following command: 

  • git branch -d branch_name

    • Replace "branch_name" with the name of the branch you want to delete.

This will delete the specified branch. Note that you cannot delete the branch you are currently on.

Pushing Changes to Github

GitHub is a web-based platform for hosting and managing Git repositories. It allows developers and teams to collaborate on code, track changes, manage projects, and contribute to open-source or private codebases.

To push your local changes to a remote repository on GitHub, follow these steps:

  • First, create a new repository on GitHub.

  • Copy the repository URL.

  • In your terminal, add the remote repository using the following command:

    git remote add origin repository_url

    Replace "repository_url" with the URL of your GitHub repository.

  • Push your changes to the remote repository using the following command:

    git push -u origin branch_name

    Replace "branch_name" with the name of the branch you want to push.

  • After pushing, you can verify that your changes are on GitHub by visiting the repository page.

Deploying your page online

You can either use Vercel or Github pages but in this case we will be using Github pages.

To deploy your page online using GitHub Pages, follow these steps:

  • Go to your GitHub repository.

  • Click on the "Settings" tab.

  • Scroll down to the "GitHub Pages" section.

  • Select the branch you want to use for GitHub Pages (usually "main").

  • Click "Save".

  • After a few moments, your page will be live at the URL provided in the GitHub Pages section.
